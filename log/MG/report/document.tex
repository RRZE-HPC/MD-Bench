\section*{Application Info}
\fakesection{Application Info}

MD-Bench is a molecular dynamics application which calculates interactions among particles and how these affect their motion.
The simulation system consists of
\begin{itemize}
    \item number of atoms (with initial state such as position \& velocity)
    \item boundary conditions for 3D space
\end{itemize}

The force of each atom is based on its accumulated interactions with neighboring atoms. In particular, MD-Bench uses the \textit{Lennard-Jones}
potential to model the potential energy among pairs of particles. The model is able to represent attractive as well as repulsive energies.

\begin{figure}[!ht]
    \centering
    \includegraphics[]{../presentation/Images/Lennard_Jones_potential_function.png}
    \caption{Lennard-Jones potential with the distance between two atoms in Ångström on the x-axis}
    \label{fig:lennardJonesPotential}
\end{figure}

In figure \ref{fig:lennardJonesPotential} the potential energy in relation to the distance between two atoms is depicted.
The formula of the potential consists of three parameters, namely:
\begin{itemize}
    \item $r$: distance between two interacting atoms
    \item $\epsilon$: dispersion energy
    \item $\sigma$: distance at which the particle-potential $V$ is zero
\end{itemize}

The figure shows that the Lennard-Jones potential is a simplified model but still describes the essential aspects of particle dynamics.
Furthermore, particles repel each other at close, attract at medium and have negligible interactions at large distances.

\newpage

\vspace*{5.5em}
MD-Bench is publicly available on GitHub via the URL \url{https://github.com/RRZE-HPC/MD-Bench}.\par
The main focus of this report is to port the main functions in \textit{force.c} to the CUDA programming model and evaluate the performance
with respect to the adapted code, which is available under the branch \textit{mucosim-cuda}.\par\par
Calculating the Lennard-Jones potential, updating the atom forces as well as positions is done in a main-loop for each time step:
\begin{figure}[!ht]
    \centering
    \begin{lstlisting}[language=Python]
        for t in range(timesteps):
            for atom in atoms:
                neighbors = neighbor.neighbors[atom]
                force = 0.0
                for neighbor in neighbors:
                    radius = calc_radius(...)
                    if radius < close_enough:
                        force += calc_force(...)
                forces[atom] += force
            update_velocity(atoms)
            update_position(atoms)
            if t % 20 == 0:
                reneighbour_atoms(atoms)
        \end{lstlisting}
        \caption{Pseudocode of the main-loop of MD-Bench}
\end{figure}

\chapter{Scaling Runs}

\section*{CUDA Programming Model}
\fakesection{CUDA Programming Model}

In order to scale the MD-Bench code efficiently on Nvidia GPUs, knowledge about the CUDA programming model
is necessary.\par
The basic execution size is a \textit{thread} which is executed by one CUDA core. Typical state-of-the art
GPUs have 4352 (RTX 2080 Ti) and up to 6912 (A100) CUDA cores.\par
Threads are grouped together as \textit{blocks}, these are then executed on Streaming Multiprocessors
(\textit{SM}). Resources on an SM are allocated on a \textit{warp}-size level (multiple of 32 threads
and up to 1024) and dispatched to execution units of the GPU.\par\par

\begin{figure}[!ht]
    \centering
    \includegraphics[width=12cm]{../presentation/Images/memory-hierarchy-in-gpus-2.png}
    \caption{Memory hierarchy in GPUs}
    \label{fig:GPUMemoryHierarchy}
\end{figure}

Figure \ref{fig:GPUMemoryHierarchy} shows the memory architectur of Nvidia GPUs such as the A100.
It consists of
\begin{itemize}
    \item Registers: Owned by a thread, not visible to other threads
    \item L1/Shared memory (SMEM): Each SM has a versatile memory which can be used as
    L1 cache or as shared memory among all threads in a CUDA block. Size ranges from
    64KB per SM (RTX 2080 Ti) to 192KB per SM (A100).
    \item Read-only memory: Each SM has an instruction cache as well as other kinds of memory
    which can be used for graphics applications.
    \item L2 cache: Shared across all SMs which means that every CUDA block can access this memory.
    Typical sizes are 5.5MB (RTX 2080 Ti) and up to 40MB (A100).
\end{itemize}

\section*{CUDA Kernels}
\fakesection{CUDA Kernels}

In order to scale performance with available CUDA cores, the code is parallelized on an atom-level.
For 131072 atoms, this means there are 131072 threads and\\
\lstinline{131702 / num_threads_per_block} number of blocks, where performance is evaluated for
\lstinline{num_threads_per_block} $\in \{1, 2, 4, 8, 16, 32, 64\}$. Each thread is going to execute
the same \textit{CUDA kernel} function in parallel on the GPU. In the context of MD-Bench,
the kernel function calculates the 3D force for a single atom with respect to its neighbour-atoms.\par
Before the execution can happen, memory has to be copied from host (CPU) to device (GPU). This means
the neighbour list as well as position data for each atom and Lennard-Jones potential parameters.
The kernel-code writes the resulting forces into GPU memory which is copied back to the host after
calculation is finished in order to proceed with other functions that are still running on CPU.\par
Inside the kernel function there is a precise way to tell which thread is currently executed.
The CUDA programming model supplies the parameters\\
\lstinline{blockIdx.x}, \lstinline{blockDim.x} and \lstinline{threadIdx.x}. A combination of these
parameters can be used as an index into the atom- and force-arrays.

\section*{Improvement IV: Force memory layout}
\fakesection{Improvement IV: Force memory layout}

The default implementation for the force memory is Structure of Arrays (\textit{SoA}). Assuming the memory was an infinitely long 1D array, then the force data would be
laying in memory like this

\begin{figure}[!ht]
    \centering
    \begin{tikzpicture}[scale=0.675]
        \def\size{0.625cm}
        \node (elem_0) at (0,0) {};
        \xdef\anchor{elem_0.east};
        \foreach \txt in {x,y,z} {
            \foreach \idx in {0,1,2} {
                \node[rectangle,draw,anchor=west,line width=0.6pt,minimum size=\size,fill=cyan!75!black] (elem_\txt_\idx) at (\anchor) {$\txt_\idx$};

                \xdef\anchor{elem_\txt_\idx.east};
            }

            \ifnum\mystrcmp{\txt}{z}=-1
                \node[anchor=west] (separator_\txt) at (elem_\txt_2.east) {$...$};
                \xdef\anchor{separator_\txt.east};
            \fi
        }
    \end{tikzpicture}
\end{figure}

Changing the force memory layout to Array of Structures (\textit{AoS}) would look like this

\begin{figure}[!ht]
    \centering
    \begin{tikzpicture}[scale=0.675]
        \def\size{0.625cm}
        \node (elem_0) at (0,0) {};
        \xdef\anchor{elem_0.east};
        \foreach \idx in {0,1,2} {
            \foreach \txt in {x,y,z} {
                \node[rectangle,draw,anchor=west,line width=0.6pt,minimum size=\size,fill=cyan!75!black] (elem_\txt_\idx) at (\anchor) {$\txt_\idx$};

                \xdef\anchor{elem_\txt_\idx.east};
            }

            \ifnum\idx<2
                \node[anchor=west] (separator_\idx) at (elem_z_\idx.east) {$...$};
                \xdef\anchor{separator_\idx.east};
            \fi
        }
    \end{tikzpicture}
\end{figure}
